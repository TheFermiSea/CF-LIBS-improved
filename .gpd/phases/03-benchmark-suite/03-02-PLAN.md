<task type="auto">
  <name>Task 1: FAISS query latency benchmark (BENCH-04)</name>
  <files>benchmarks/bench_faiss.py</files>
  <action>
    Create benchmarks/bench_faiss.py:

    1. Import faiss (handle ImportError for both faiss-cpu and faiss-gpu gracefully)
    2. Import common utilities from benchmarks.common
    3. Define database_sizes:
       - Full mode (default): [100_000, 1_000_000, 10_000_000, 100_000_000]
       - Small mode (--small flag): [1_000, 10_000, 100_000]
    4. Define vector dimension d = 30 (matching SpectralEmbedder PCA dimension)
    5. Define n_queries = 100 (batch of 100 queries per measurement)
    6. Define k = 10 (10 nearest neighbors)

    For each database_size:
      a. Generate random float32 vectors: db_vectors = np.random.randn(database_size, d).astype(np.float32)
         (L2-normalize to match SpectralEmbedder convention)
      b. Generate query vectors: queries = np.random.randn(n_queries, d).astype(np.float32) (L2-normalized)

      c. CPU IndexFlatL2:
         - Build index, time the build
         - Search k=10 neighbors, time the search (n_runs=10 repetitions)
         - Record: build_time, search_time_mean, search_time_std, per_query_latency
         - Save ground-truth neighbors for recall computation

      d. CPU IndexIVFFlat (nlist=int(sqrt(database_size)), nprobe=10):
         - Train on db_vectors (or subset if >1M), time the train+build
         - Search k=10, time the search
         - Compute recall@1 vs exact (FlatL2) neighbors
         - Record: build_time, search_time_mean, search_time_std, recall_at_1

      e. GPU IndexFlatL2 (if faiss-gpu available):
         - Use faiss.StandardGpuResources()
         - faiss.index_cpu_to_gpu(res, 0, cpu_flat_index)
         - Time search (n_runs=10)
         - Record same metrics as CPU
         - Handle OOM for 100M case gracefully (skip and note)

      f. GPU IVFFlat (if faiss-gpu available):
         - Transfer trained IVF index to GPU
         - Time search
         - Compute recall@1
         - Handle OOM gracefully

    7. Output JSON:
       ```json
       {
         "benchmark": "faiss_query_latency",
         "hardware": {...},
         "parameters": {"db_sizes": [...], "d": 30, "k": 10, "n_queries": 100},
         "results": [
           {
             "db_size": N,
             "cpu_flat_build_s": ...,
             "cpu_flat_search_ms_mean": ..., "cpu_flat_search_ms_std": ...,
             "cpu_flat_per_query_us": ...,
             "cpu_ivf_build_s": ...,
             "cpu_ivf_search_ms_mean": ..., "cpu_ivf_search_ms_std": ...,
             "cpu_ivf_recall_at_1": ...,
             "gpu_flat_search_ms_mean": ..., "gpu_flat_search_ms_std": ...,
             "gpu_flat_per_query_us": ...,
             "gpu_ivf_search_ms_mean": ..., "gpu_ivf_search_ms_std": ...,
             "gpu_ivf_recall_at_1": ...,
             "gpu_available": true/false,
             "gpu_oom": true/false
           }
         ]
       }
       ```
    8. Runnable as: `python benchmarks/bench_faiss.py --output benchmarks/results/faiss_results.json`
    9. Flags: --small (reduced sizes), --cpu-only (skip GPU), --output PATH

    Memory safety: For 100M vectors at d=30 float32, that is 12 GB. Check available GPU memory before attempting GPU index and skip if insufficient. Use numpy memory-mapped arrays or chunked generation for the 100M case to avoid CPU OOM.
  </action>
  <verify>
    1. `python benchmarks/bench_faiss.py --small --cpu-only --output /tmp/test_faiss.json` produces valid JSON
    2. CPU FlatL2 recall@1 = 1.0 (exact search is exact)
    3. CPU IVFFlat recall@1 > 0.8 (approximate search should be decent)
    4. Latency increases with database_size (basic sanity)
    5. Build time increases with database_size
    6. Script handles missing faiss-gpu gracefully (gpu_available: false in output)
  </verify>
  <done>benchmarks/bench_faiss.py produces valid JSON with latency and recall data for all index types and database sizes, handles GPU unavailability and OOM gracefully</done>

<task type="auto">
  <name>Task 2: Batch forward model throughput benchmark (BENCH-05)</name>
  <files>benchmarks/bench_batch_forward.py</files>
  <action>
    Create benchmarks/bench_batch_forward.py:

    1. Import batch_forward_model from cflibs.manifold.batch_forward
    2. Import common utilities from benchmarks.common
    3. Define batch_sizes:
       - Full mode: [1, 5, 10, 50, 100, 500, 1000, 5000, 10000]
       - Small mode (--small): [1, 5, 10, 50, 100]
    4. Define N_lines = 50 (representative line count for multi-element LIBS)
    5. Define N_wl = 4096 (wavelength grid points, matching manifold standard)

    For each batch_size:
      a. Generate synthetic plasma parameters:
         - T: uniform random in [8000, 15000] K, shape (batch_size,)
         - n_e: log-uniform in [1e15, 1e17] cm^-3, shape (batch_size,)
         - C: random simplex composition for 5 elements, shape (batch_size, 5)
      b. Generate synthetic atomic data:
         - wavelengths: 50 lines uniformly in [200, 600] nm
         - A_ki: log-uniform in [1e6, 1e9] s^-1
         - E_k: uniform in [0, 10] eV
         - g_k: random integers 1..10
         - IP: [7.9, 15.0, 7.7, 6.8, 5.4] eV (Fe, Cu-like, etc.)
         - element_indices: assign lines to elements
      c. Generate wavelength grid: wl = np.linspace(200, 600, N_wl) nm

      d. CPU sequential baseline: time a Python for-loop calling single_spectrum for each batch element
         - This represents the current non-GPU approach
         - Record: total_time, per_spectrum_time
      e. GPU batch: time batch_forward_model(T_batch, ne_batch, C_batch, atomic_data, wl_grid)
         - Include device_put time in a separate measurement
         - Record: total_time_with_transfer, total_time_compute_only, per_spectrum_time

    6. Compute throughput: spectra_per_sec = batch_size / total_time
    7. Compute memory usage estimate: batch_size * N_wl * N_lines * 8 bytes (float64 intermediate)

    8. Output JSON:
       ```json
       {
         "benchmark": "batch_forward_model",
         "hardware": {...},
         "parameters": {"batch_sizes": [...], "n_lines": 50, "n_wl": 4096, "n_elements": 5, "n_runs": 10},
         "results": [
           {
             "batch_size": B,
             "cpu_total_time_ms_mean": ..., "cpu_total_time_ms_std": ...,
             "cpu_per_spectrum_ms": ...,
             "cpu_throughput_spectra_per_sec": ...,
             "gpu_total_time_ms_mean": ..., "gpu_total_time_ms_std": ...,
             "gpu_per_spectrum_ms": ...,
             "gpu_throughput_spectra_per_sec": ...,
             "gpu_transfer_time_ms": ...,
             "speedup": ...,
             "estimated_memory_gb": ...
           }
         ]
       }
       ```
    9. Runnable as: `python benchmarks/bench_batch_forward.py --output benchmarks/results/batch_forward_results.json`
    10. Flags: --small, --cpu-only, --output PATH
    11. Handle OOM for large batch sizes: catch RuntimeError/jax.errors and record "oom": true

    IMPORTANT: The CPU baseline must be a Python for-loop calling the single-spectrum function, NOT a vectorized NumPy implementation. This reflects the actual current CPU approach. The GPU uses vmap over the batch dimension.

    For batch_size >= 5000, reduce n_runs to 3 (each run takes significant time).
  </action>
  <verify>
    1. `python benchmarks/bench_batch_forward.py --small --cpu-only --output /tmp/test_batch.json` produces valid JSON
    2. CPU per_spectrum_ms is roughly constant across batch sizes (loop overhead is per-element)
    3. Throughput values are positive and computed correctly
    4. Memory estimate is computed: batch_size * 4096 * 50 * 8 / 1e9 GB
    5. OOM handling works (set --max-batch 10 and verify graceful skip for larger sizes)
    6. JSON output has entries for each batch_size tested
  </verify>
  <done>benchmarks/bench_batch_forward.py produces valid JSON with throughput, timing, and memory data for all batch sizes, handles OOM gracefully, CPU baseline uses sequential loop</done>
